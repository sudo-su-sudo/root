## Decision Context Preservation

### Decision: Learning vs. Rule-Based
**When**: During Timeline Alpha development
**Context**: Need for autonomous action without constant user input
**Options Considered**:
1. Pure rule-based system (transparent but rigid)
2. Learning system (adaptive but less transparent)
3. Hybrid approach (complex but balanced)

**Decision Made**: Learning system with meta-reasoning
**Rationale**: 
- Autonomous action requires understanding preferences
- Meta-reasoning provides transparency about uncertainty
- User can check in when confidence is low

**Alternative Preserved**: `examples.py` shows basic rule-based approach

### Decision: Persistence Strategy
**When**: During Timeline Beta development
**Context**: Mobile browser sessions are unreliable
**Options Considered**:
1. Session-only storage (simple but data loss on close)
2. LocalStorage (browser-based but limited)
3. Database persistence (complex but reliable)

**Decision Made**: Database persistence with SQLite
**Rationale**:
- Mobile users need data to survive browser closure
- Phone restarts shouldn't lose learning history
- Server-side storage enables future multi-device sync

**Alternative Preserved**: Documentation explains other options

### Decision: Documentation Philosophy
**When**: During Timeline Delta development
**Context**: Complex system with many concepts to explain
**Options Considered**:
1. Technical API documentation only
2. User-focused guides only
3. Comprehensive multi-level documentation

**Decision Made**: Comprehensive educational documentation
**Rationale**:
- Users have different knowledge levels
- Complex concepts need careful explanation
- "Why" is as important as "how"

**Implementation**: 18 markdown files covering all aspects

## Inferring Complete Context

For any given point in time, you can infer complete context by:

1. **Check the commit date**: When did this happen?
2. **Check the branch**: Which timeline does it belong to?
3. **Read the commit message**: What was changed and why?
4. **Check related branches**: What else was happening in parallel?
5. **Read contemporaneous docs**: What was understood at that time?
6. **Check merge commits**: How did timelines influence each other?

## Principles for Timeline Maintenance

1. **Acknowledge Parallel Work**: When documenting a branch, note what else was happening
2. **Preserve Context**: Document what was known at the time, not what you know now
3. **Cross-Reference**: Link related work across timelines
4. **Explain Decisions**: Document why choices were made, not just what was done
5. **Track Influence**: Note how one timeline informed another
6. **Maintain Chronology**: Keep time-based ordering even if branches merged later
